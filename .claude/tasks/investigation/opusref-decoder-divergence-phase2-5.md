## Objective

Root-cause and fix the first `FuzzOpusrefDecoderDifferential` finding:
Pure Go and libopus both accept a 12 kHz mono packet, but decoded output differs
grossly.

## Reproducer

Fuzz target:

```text
go test -count=1 -tags opusref -run '^FuzzOpusrefDecoderDifferential$' -v .
```

The minimized failing fuzz input generated by Go fuzzing was:

```go
[]byte("\x01(\xa7\x19\xff\xff\x00\x00\xed\x99\xf1\xb2\xd0\x1e,h\xb2\xd7\xc7\xdb\xc3\xc8w\x0e3S\x12\x16g\xb6qON\x88b5J\x13BQq\x88\xdeKvw\"R$")
```

Decoded fields:

The original generated corpus file was ignored by `.gitignore` (`testdata/`) and
was removed after recording this literal.

## Root Cause

The SILK entropy decode was not the problem. Instrumented libopus tracing showed
that the SILK indices, gains, NLSF, pitch/LTP parameters, pulses, excitation,
and core SILK output matched the Go decoder.

The divergence came from a trailing CELT-to-SILK leading-redundancy frame carried
after the SILK stream. libopus decodes that redundant CELT frame and applies it
unless the previous decoded mode was SILK-only without previous redundancy. The
Go decoder only applied leading redundancy when `prevMode == CELTOnly`, so a
first packet in a stream discarded redundancy that libopus applies.

The fix updates both SILK-only and hybrid decode paths to apply leading
redundancy whenever the previous mode is not SILK-only.

## Implemented Fix

- Added `opusref_decoder_divergence_test.go` with
  `TestOpusrefDecoderDifferentialMinimized12kSILK`.
- Updated `decodeSILKPacket` and `decodeHybridPacket` leading-redundancy
  application from `prevMode == CELTOnly` to `prevMode != SILKOnly`.
- Adjusted `FuzzOpusrefDecoderDifferential` gross-output oracle so random
  accepted packets with very large but matching finite output do not fail solely
  due to absolute peak level.
