## Objective

Add a local-only `opusref` differential decoder fuzz target that feeds the same
bounded packet inputs to the Pure Go decoder and libopus 1.6.1 reference decoder,
then records accept/reject and output-divergence findings for Phase 2-5 fixes.

This is a diagnostic conformance detector, not a nightly CI target by default.
It requires `-tags opusref`, CGO, and a local libopus installation.

## Scope

Implement one bounded fuzz target in the root package, likely behind
`//go:build opusref`, for single-stream packets first.

Recommended target name:

```text
FuzzOpusrefDecoderDifferential
```

Recommended file:

```text
opusref_differential_fuzz_test.go
```

## Required behavior

1. Select only valid public decoder constructors:
   - sample rates: `8000`, `12000`, `16000`, `24000`, `48000`
   - channels: mono/stereo
2. Bound input size and decoder work:
   - reject empty or very large fuzz inputs early
   - cap packet bytes to a small fixed size first, e.g. 512 or 1275 bytes
   - use `MaxFrameSize * rate / 48000` as the max samples-per-channel decode size
3. Decode the same packet with:
   - Pure Go `NewDecoder(rate, channels).DecodeFloat32` or `Decode`
   - `internal/cgoref.NewDecoder(rate, channels).DecodeFloat(packet, maxSPC)`
4. Compare outcomes:
   - both reject: accepted result, no failure
   - Go accepts / libopus rejects: record as a differential finding
   - Go rejects / libopus accepts: record as a differential finding
   - both accept: compare decoded length and basic output sanity
5. For both-accepted packets, avoid a strict sample-exact oracle initially.
   Use bounded diagnostics:
   - same sample count per channel is required
   - outputs must be finite
   - gross RMS/peak divergence can fail only with a clear threshold and comment
     explaining why known PLC/random-packet behavior is not being overclaimed
6. If `FinalRange` can be read from libopus after successful decode, compare it
   only where the packet mode is expected to be bit-exact. Otherwise log or
   structure the helper for future tightening.

## Important guardrails

- Do not add this target to the normal fuzz workflow unless explicitly approved.
  `opusref` fuzzing is local-only because it requires CGO/libopus and may expose
  reference-vs-Go conformance differences that need triage before automation.
- Do not weaken production decoder validation to match libopus during the fuzz
  target implementation. Any discovered divergence should become a Phase 2-5
  root-cause fix with a committed minimized seed.
- Keep wording and tool use scoped to local authorized repository testing.
- Do not treat every sample-level difference as a bug. Start with accept/reject,
  duration, finiteness, and large-output-divergence detection.

## Seeds

Include deterministic seeds for:

- empty/malformed packet shapes
- a Pure-Go CELT packet generated by `NewEncoder(48000, 1/2, ApplicationAudio)`
- a Pure-Go SILK-only voice packet generated at 8/12/16 kHz
- a Pure-Go hybrid voice packet generated at 24/48 kHz
- one code-3/multi-frame packet if compact generation is straightforward

Seeds must be generated deterministically at test startup or stored as compact
literal packets. Prefer generated seeds when setup is cheap and deterministic.

## Qualification

Required local qualification before adoption:

```text
go test -count=1 -tags opusref -run '^FuzzOpusrefDecoderDifferential$' -v .
go test -run='^$' -tags opusref -fuzz='^FuzzOpusrefDecoderDifferential$' -fuzztime=30m -fuzzminimizetime=10x -parallel=1 -timeout=31m -v .
go vet ./...
go test -count=1 ./...
go test -count=1 -tags opusref ./...
```

If the fuzz run finds a real divergence, stop and create a Phase 2-5 fix
iteration instead of masking it in the target.
